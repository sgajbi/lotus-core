# RFC-POSITION-DUAL-LEG-ALIGNMENT-01 Proposed Alignment of BUY, SELL, DIVIDEND, and INTEREST with the Dual-Leg Accounting Model

## 1. Purpose

This RFC proposes required changes to the existing transaction specifications for:

* `BUY`
* `SELL`
* `DIVIDEND`
* `INTEREST`

so that they conform to the canonical dual-leg accounting model defined in:

* `shared/13-dual-leg-accounting-and-cash-adjustment-model.md`

This RFC is a **change RFC**. It does not replace the individual transaction RFCs.
It defines the mandatory updates those RFCs must incorporate.

Initial rollout scope in this RFC is `BUY`, `SELL`, `DIVIDEND`, and `INTEREST`.
However, the dual-leg accounting pattern defined here is generic and must be used by any future transaction type that carries both product semantics and cash impact.

---

## 2. Summary of Required Change

The following position-level transaction types must be treated as **dual-leg economic events whenever a cash movement exists**:

* `BUY`
* `SELL`
* `DIVIDEND`
* `INTEREST`

Each such event must consist of:

1. a **product leg** using the original transaction type, and
2. a linked **cash leg** using `transaction_type = ADJUSTMENT`

This is mandatory regardless of whether the cash leg is:

* generated by the engine, or
* supplied by upstream and linked later

Once linked, downstream behavior must be identical.

This equivalence rule also applies to future transaction types once they adopt the dual-leg model.

---

## 3. Canonical Change Being Introduced

## 3.1 Existing logical problem being corrected

The existing transaction specifications describe cash behavior, but they do not fully elevate the cash leg to a first-class, canonical transaction representation shared across all position-level flows.

This creates a risk of:

* inconsistent cash representation
* duplicated cash interpretation
* ambiguity about whether the product leg or cash leg is authoritative for cash
* downstream double counting in performance and contribution

## 3.2 New required model

The new required model is:

* **Product leg** retains product semantics
* **Cash leg (`ADJUSTMENT`)** owns cash semantics
* both are part of one economic event
* both are always linked
* cash movement for downstream balance and performance purposes is sourced from the cash leg

---

## 4. Cross-Cutting Requirements Added to All Four Transaction Types

The following requirements must be added to all four RFCs.

## 4.1 Mandatory dual-leg rule

For `BUY`, `SELL`, `DIVIDEND`, and `INTEREST`:

* if the transaction has a cash impact, it must be represented as a dual-leg event
* the product leg keeps the original transaction type
* the cash leg uses `transaction_type = ADJUSTMENT`

## 4.2 Mandatory linkage rule

Both legs must share:

* `economic_event_id`
* `linked_transaction_group_id`

Both legs must be directly linkable using:

* `originating_transaction_id`
* `linked_cash_transaction_id`
* `link_type`
* `reconciliation_key` where applicable

## 4.3 Mandatory cash-leg ownership rule

The `ADJUSTMENT` cash leg becomes the authoritative source for:

* cash balance movement
* cash ledger update
* performance cash flow
* contribution cash flow input

The product leg may still carry expected cash fields, but only for:

* validation
* explainability
* reconciliation

The product leg must not be the authoritative source of cash mutation when a valid cash leg exists.

## 4.4 Mandatory no-double-count rule

Downstream systems must not derive a second cash flow from the product leg if a linked `ADJUSTMENT` cash leg exists.

This requirement must be stated explicitly in all four RFCs.

## 4.5 Mandatory engine/upstream equivalence rule

The RFCs must state explicitly that:

* engine-generated cash leg
* upstream-provided cash leg

produce identical downstream meaning once linked.

## 4.6 Mandatory `ADJUSTMENT` metadata rule

Every linked cash leg must include:

* `transaction_type = ADJUSTMENT`
* `originating_transaction_id`
* `originating_transaction_type`
* `adjustment_reason`
* `economic_event_id`
* `linked_transaction_group_id`
* `link_type`


## 4.7 Mandatory net-cash-leg amount rule

For `BUY`, `SELL`, `DIVIDEND`, and `INTEREST`, the linked `ADJUSTMENT` cash leg must use the following standard:

* the cash-leg amount must equal the **net cash amount** of the event
* the cash-leg direction must be the **actual cash movement direction**
* the cash-leg sign must be **opposite to the product leg’s cash-impact representation**

### Per transaction type

* `BUY`

  * `ADJUSTMENT.amount = net settlement cash`
  * `ADJUSTMENT.direction = OUTFLOW`

* `SELL`

  * `ADJUSTMENT.amount = net proceeds`
  * `ADJUSTMENT.direction = INFLOW`

* `DIVIDEND`

  * `ADJUSTMENT.amount = net dividend cash`
  * `ADJUSTMENT.direction = INFLOW`

* `INTEREST`

  * `ADJUSTMENT.amount = net interest cash`
  * `ADJUSTMENT.direction = INFLOW` for interest income
  * `ADJUSTMENT.direction = OUTFLOW` for interest expense

### Required RFC updates

Each of the four transaction RFCs must explicitly state that:

* the product leg may carry expected cash fields
* the linked `ADJUSTMENT` cash leg is booked using the **net amount**
* the linked `ADJUSTMENT` cash leg follows the **actual cash direction**
* the linked `ADJUSTMENT` cash leg has the **opposite sign** to the product leg’s cash-impact representation
* downstream cash and performance engines must consume the `ADJUSTMENT` amount as the authoritative cash flow

This is mandatory across all four RFCs.

---

## 5. Required Changes to RFC-BUY-01

## 5.1 Business-definition change

The `BUY` RFC must explicitly state:

* `BUY` is the **product leg**
* whenever settlement cash is required, a linked `ADJUSTMENT` **cash leg** is mandatory

## 5.2 Data-model change

Add explicit linked cash-leg fields to the `BUY` RFC and product-leg model language:

* `has_linked_cash_leg: bool`
* `linked_cash_transaction_id: str | None`
* `expected_cash_leg_transaction_type = ADJUSTMENT`
* `originating_transaction_type = BUY` on the cash leg
* `adjustment_reason = BUY_SETTLEMENT` on the cash leg

Add explicit cash-leg model reference requirements:

* `cash_instrument_id`
* `cash_account_id`
* `movement_direction = OUTFLOW`
* `cash_amount_local`
* `cash_amount_base`

## 5.3 Processing change

The `BUY` processing flow must explicitly require:

* create / link `ADJUSTMENT` cash leg after deriving settlement cash
* persist the product leg and cash leg as one linked economic event
* park the event if upstream mode is selected and the cash leg has not yet arrived, where policy requires complete pairing

## 5.4 Cash-rule change

The `BUY` RFC must explicitly state:

* the cash leg is the source of truth for reducing cash
* the product leg’s cash fields are informative only when a cash leg exists

## 5.5 Query/output change

The `BUY` RFC must require downstream query visibility for:

* product leg
* linked `ADJUSTMENT` cash leg
* grouped economic-event view containing both

## 5.6 Test change

The `BUY` RFC test matrix must add:

* `BUY` + engine-generated `ADJUSTMENT`
* `BUY` + upstream-provided `ADJUSTMENT`
* missing required cash leg
* duplicate cash leg prevention
* cash sourced from `ADJUSTMENT` only
* no double-count in performance flow

## 6. Required Changes to RFC-SELL-01

## 6.1 Business-definition change

The `SELL` RFC must explicitly state:

* `SELL` is the **product leg**
* whenever settlement cash is received, a linked `ADJUSTMENT` **cash leg** is mandatory

## 6.2 Data-model change

Add explicit linked cash-leg fields to the `SELL` RFC and product-leg model language:

* `has_linked_cash_leg: bool`
* `linked_cash_transaction_id: str | None`
* `expected_cash_leg_transaction_type = ADJUSTMENT`
* `originating_transaction_type = SELL` on the cash leg
* `adjustment_reason = SELL_SETTLEMENT` on the cash leg

Add explicit cash-leg model reference requirements:

* `cash_instrument_id`
* `cash_account_id`
* `movement_direction = INFLOW`
* `cash_amount_local`
* `cash_amount_base`

## 6.3 Processing change

The `SELL` processing flow must explicitly require:

* create / link `ADJUSTMENT` cash leg after deriving settlement proceeds
* persist the product leg and cash leg as one linked economic event
* prevent independent cash posting that is not linked back to the `SELL`

## 6.4 Cash-rule change

The `SELL` RFC must explicitly state:

* the cash leg is the source of truth for increasing cash
* the product leg’s proceeds fields do not independently mutate cash when a linked cash leg exists

## 6.5 Query/output change

The `SELL` RFC must require downstream query visibility for:

* product leg
* linked `ADJUSTMENT` cash leg
* grouped economic-event view containing both

## 6.6 Test change

The `SELL` RFC test matrix must add:

* `SELL` + engine-generated `ADJUSTMENT`
* `SELL` + upstream-provided `ADJUSTMENT`
* missing required cash leg
* duplicate cash leg prevention
* proceeds represented in product leg, cash mutation represented in cash leg
* no double-count in performance flow

## 7. Required Changes to RFC-DIVIDEND-01

## 7.1 Business-definition change

The `DIVIDEND` RFC must explicitly state:

* `DIVIDEND` is the **product leg**
* when cash is paid, a linked `ADJUSTMENT` **cash leg** is mandatory

## 7.2 Data-model change

Add explicit linked cash-leg fields to the `DIVIDEND` RFC and product-leg model language:

* `has_linked_cash_leg: bool`
* `linked_cash_transaction_id: str | None`
* `expected_cash_leg_transaction_type = ADJUSTMENT`
* `originating_transaction_type = DIVIDEND` on the cash leg
* `adjustment_reason = DIVIDEND_SETTLEMENT` on the cash leg

Add explicit cash-leg model reference requirements:

* `cash_instrument_id`
* `cash_account_id`
* `movement_direction = INFLOW`
* `cash_amount_local`
* `cash_amount_base`

## 7.3 Processing change

The `DIVIDEND` processing flow must explicitly require:

* create / link `ADJUSTMENT` cash leg after deriving net cash receipt
* preserve gross income, tax, and net income on the product leg
* use the cash leg as the actual cash balance mutation

## 7.4 Cash-rule change

The `DIVIDEND` RFC must explicitly state:

* income semantics remain on the product leg
* actual cash receipt for balance and performance purposes comes from the `ADJUSTMENT` cash leg

## 7.5 Query/output change

The `DIVIDEND` RFC must require downstream query visibility for:

* product leg with income/tax decomposition
* linked `ADJUSTMENT` cash leg
* grouped economic-event view containing both

## 7.6 Test change

The `DIVIDEND` RFC test matrix must add:

* `DIVIDEND` + engine-generated `ADJUSTMENT`
* `DIVIDEND` + upstream-provided `ADJUSTMENT`
* gross/net/tax on product leg, cash on cash leg
* missing required cash leg
* duplicate cash leg prevention
* no double-count in performance flow

## 8. Required Changes to RFC-INTEREST-01

## 8.1 Business-definition change

The `INTEREST` RFC must explicitly state:

* `INTEREST` is the **product leg** for position-level / instrument-level interest flows
* when cash is settled, a linked `ADJUSTMENT` **cash leg** is mandatory

## 8.2 Data-model change

Add explicit linked cash-leg fields to the `INTEREST` RFC and product-leg model language:

* `has_linked_cash_leg: bool`
* `linked_cash_transaction_id: str | None`
* `expected_cash_leg_transaction_type = ADJUSTMENT`
* `originating_transaction_type = INTEREST` on the cash leg
* `adjustment_reason = INTEREST_SETTLEMENT` or `INTEREST_CHARGE_SETTLEMENT` on the cash leg

Add explicit cash-leg model reference requirements:

* `cash_instrument_id`
* `cash_account_id`
* `movement_direction = INFLOW` for income
* `movement_direction = OUTFLOW` for expense
* `cash_amount_local`
* `cash_amount_base`

## 8.3 Processing change

The `INTEREST` processing flow must explicitly require:

* create / link `ADJUSTMENT` cash leg after deriving net interest settlement amount
* retain income/expense semantics on the product leg
* apply actual cash mutation only through the cash leg

## 8.4 Cash-rule change

The `INTEREST` RFC must explicitly state:

* product leg owns interest recognition semantics
* cash leg owns cash debit / credit semantics
* this applies to both interest income and interest expense

## 8.5 Query/output change

The `INTEREST` RFC must require downstream query visibility for:

* product leg
* linked `ADJUSTMENT` cash leg
* grouped economic-event view containing both

## 8.6 Test change

The `INTEREST` RFC test matrix must add:

* interest income + engine-generated `ADJUSTMENT`
* interest income + upstream-provided `ADJUSTMENT`
* interest expense + engine-generated `ADJUSTMENT`
* missing required cash leg
* duplicate cash leg prevention
* no double-count in performance flow

---

## 9. Required New Shared Definitions Across All Four RFCs

The following common terms must be added consistently to each RFC.

## 9.1 Product-leg definition

The original transaction type (`BUY`, `SELL`, `DIVIDEND`, `INTEREST`) is the product leg.

## 9.2 Cash-leg definition

A linked `ADJUSTMENT` transaction is the cash leg.

## 9.3 Mandatory cash-leg fields

Every RFC must reference the requirement that the linked `ADJUSTMENT` cash leg includes:

* `transaction_type = ADJUSTMENT`
* `cash_instrument_id`
* `cash_account_id`
* `cash_amount_local`
* `cash_amount_base`
* `movement_direction`
* `originating_transaction_id`
* `originating_transaction_type`
* `adjustment_reason`
* `economic_event_id`
* `linked_transaction_group_id`
* `link_type`

## 9.4 Mandatory event-group output

Each RFC must require a grouped output or queryable join that can show:

* product leg
* cash leg
* shared economic-event identity

This is required for support, reconciliation, and reporting.

---

## 10. Required Downstream Consumption Changes

The transaction RFCs must be updated so downstream expectations are explicit.

## 10.1 Position engine

Consumes the product leg.

## 10.2 Cash ledger

Consumes the `ADJUSTMENT` cash leg.

## 10.3 Performance engine

Consumes:

* product leg for non-cash business semantics
* `ADJUSTMENT` cash leg for cash-flow event sourcing

## 10.4 Reporting

Must be able to report:

* separate legs, and
* grouped economic event

## 10.5 Reconciliation

Must validate:

* product-leg expected cash
* actual `ADJUSTMENT` cash leg
* linkage consistency

---

## 11. Required Validation Enhancements

All four RFCs must add explicit validation that:

* a required `ADJUSTMENT` cash leg exists whenever the transaction has a cash impact
* the cash leg amount matches expected economic cash within tolerance
* the cash leg direction matches the product semantics
* the cash leg references the correct originating transaction type
* duplicate `ADJUSTMENT` legs for the same event are prevented

---

## 12. Required Test Enhancements Across All Four RFCs

A common minimum test set must be added to all four RFCs:

* valid product leg with engine-generated `ADJUSTMENT`
* valid product leg with upstream-provided `ADJUSTMENT`
* missing required `ADJUSTMENT`
* mismatched cash amount between product expectation and cash leg
* mismatched direction between product and cash leg
* duplicate cash leg prevention
* stable linkage fields
* replay without duplicating product or cash leg
* no double counting of cash in performance and contribution flows

---

## 13. Migration / Implementation Guidance

The change should be implemented as a controlled RFC-alignment update, not as an informal convention.

Recommended implementation order:

1. add shared dual-leg standard
2. update shared modeling guidance to include `ADJUSTMENT`
3. update `BUY`
4. update `SELL`
5. update `DIVIDEND`
6. update `INTEREST`
7. add characterization tests for current correct behavior
8. add dual-leg conformance tests
9. update downstream read models / grouped-event views

---

## 14. Recommended Naming Decision

The recommended canonical generic cash-leg transaction type is:

* `ADJUSTMENT`

This should be adopted unless there is a deliberate platform-wide naming decision to use something more specific such as:

* `CASH_ADJUSTMENT`
* `SETTLEMENT_ADJUSTMENT`

At present, `ADJUSTMENT` is acceptable and suitable, provided the mandatory reason/origin fields are always present.

---

## 15. Completion Criteria

This alignment RFC is satisfied only when:

* shared dual-leg documentation exists
* all four RFCs explicitly reference the shared dual-leg standard
* all four RFCs define the original transaction as the product leg
* all four RFCs define linked `ADJUSTMENT` as the mandatory cash leg when cash exists
* all four RFCs explicitly state the cash leg is the cash source of truth
* all four RFCs add no-double-count language
* all four RFCs add required validation, query, and test changes
* downstream systems can consume grouped product+cash events consistently

---

## 16. Final Authoritative Change Statement

This RFC introduces a mandatory cross-cutting change to `BUY`, `SELL`, `DIVIDEND`, and `INTEREST`:

Whenever these position-level transactions have a cash impact, they must be implemented as dual-leg economic events consisting of:

* the original transaction type as the **product leg**
* a linked `ADJUSTMENT` transaction as the **cash leg**

This becomes the required accounting model for these transaction types going forward.

The same architecture and invariants are intended as the platform-standard pattern for future dual-leg transaction types.
