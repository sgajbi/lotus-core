# Database Migrations

## Purpose

`lotus-core` uses Alembic for schema migration governance.

Schema changes are part of the product contract here because portfolio, transaction, valuation,
supportability, and replay state all depend on deterministic database shape.

## Standard workflow

### 1. Change the SQLAlchemy models

Most schema work starts in:

- [database_models.py](../src/libs/portfolio-common/portfolio_common/database_models.py)

### 2. Generate a migration

For a local Python environment:

```bash
python -m alembic revision --autogenerate -m "add descriptive change"
```

For the app-local Docker workflow:

```bash
docker compose run --rm migration-runner alembic revision --autogenerate -m "add descriptive change"
```

### 3. Review the generated migration

Autogenerated migrations are a starting point, not proof of correctness.

Check for:

- unintended drops or type changes
- missing indexes or constraints
- safe nullable to non-null transitions
- deterministic forward behavior for existing data

### 4. Apply the migration

Repo-native local apply:

```bash
make migration-apply
```

Equivalent Alembic command:

```bash
python -m alembic upgrade head
```

Docker workflow:

```bash
docker compose run --rm migration-runner alembic upgrade head
```

### 5. Run migration validation

Smoke-check the migration contract:

```bash
make migration-smoke
```

For a broader slice, include the feature-lane gate:

```bash
make ci-local
```

## Runtime notes

- `migration-runner` is part of the app-local compose flow in `docker-compose.yml`
- the stack expects migrations to reach `head` before dependent services fully operate
- migration drift in `lotus-core` can surface as query-service, control-plane, or worker failures

## Production posture

1. prefer forward-only fixes over relying on downgrade in production
2. commit model changes and the matching Alembic revision together
3. treat data backfill and runtime compatibility as part of the change, not an afterthought
4. document operationally meaningful schema changes in the relevant RFC or architecture note when
   they alter supportability, lineage, or contract semantics
